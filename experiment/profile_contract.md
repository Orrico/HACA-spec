### 2.3 Profile Contract

A Cognitive Profile specification is a normative document that instantiates the operational contract for a specific profile. Every profile specification must define the following items. HACA-Arch delegates these decisions to the profile layer because they are operationally sensitive and context-dependent: the correct values differ between zero-autonomy and supervised-autonomy deployments, between regulated and research contexts, and across different host environments. A profile specification that omits any of the mandatory items below is incomplete and must not be used as the basis for a compliant deployment.

**Drift Framework — mandatory per drift type:**

1. **Inference Drift** — detection criteria (what constitutes a contradiction against Behavioral Probes), tolerance thresholds (when a contradiction is classified as Degraded vs. Critical), and response procedures (what the SIL does upon receiving an Inference Drift signal, including whether the cycle is halted, retried, or escalated).
2. **Semantic Drift** — detection criteria (what constitutes divergence against Semantic Probes), tolerance thresholds (the acceptable deviation bounds before signaling), and response procedures. Under HACA-Core, any detected divergence is an anomaly by architectural invariant; the profile must still define the response procedure. Under HACA-Evolve, the profile must additionally define the cross-referencing procedure for Structural Consolidation Proposals.
3. **Identity Drift** — detection criteria (which structural hash mismatches are classified as Degraded vs. Critical), tolerance thresholds (whether any mismatch is immediately Critical or whether localized mismatches may be treated as Degraded), and response procedures.
4. **Evolutionary Drift** — detection criteria (the method for measuring cumulative structural distance across the integrity chain), tolerance thresholds (the maximum authorized distance before requiring Operator review), and response procedures.

**Memory — mandatory:**

5. **Garbage collection criteria** — what constitutes "expired or superseded" mnemonic content eligible for removal during the Sleep Cycle. The profile must define expiry rules for episodic records, supersession rules for replaced knowledge entries, and any content categories that are additionally exempt. The following categories are architecturally exempt from garbage collection in all profiles and must never be removed: the Imprint Record, the Genesis Omega, all Integrity Log entries, the active Behavioral Probes, and the active Semantic Probes. A profile that defines garbage collection criteria that would remove any of these categories is non-compliant.

**Structural Evolution — mandatory:**

6. **Structural Consolidation Proposal governance** — the criteria under which the CPE may generate a Structural Consolidation Proposal (what accumulated semantic content qualifies as a candidate for structural promotion), and the criteria under which the SIL approves or rejects a proposal. Each proposal generated by the CPE carries a unique proposal ID, recorded in the Integrity Log at approval or rejection time. Under HACA-Core, an out-of-band authorization signal must reference the proposal ID it authorizes; the SIL rejects signals that do not reference a valid pending proposal ID. Proactive authorization — delivered before the proposal exists — is not supported; authorization is always post-proposal. Under HACA-Core, the profile must define what constitutes a valid authorization signal and must include at least one mechanism that does not depend on the entity's active Cognitive Cycle — for example, a direct instruction delivered through the Operator Channel — to ensure that structural changes that would otherwise be blocked by existing Behavioral Probe validation can still be authorized. An authorization signal delivered out-of-band through the Operator Channel is verified by the SIL directly without routing through the CPE or triggering Inference Drift detection. Under HACA-Evolve, the profile must define the operational authorization scope as a set of capability boundaries — anchored in the Imprint Record and updatable via Endure — within which the SIL may autonomously approve proposals without Operator involvement. The scope definition must be precise enough to serve as a deterministic yes/no criterion for any given proposal: if a proposal cannot be unambiguously classified as inside or outside the scope, the SIL must reject it and escalate to the Operator. A proposal that would modify the operational authorization scope itself is subject to an additional constraint: it must itself fall within the current authorization scope, or require explicit Operator authorization regardless of the active profile — the entity may not autonomously expand the boundaries of its own autonomy.
7. **Probe revision governance** — the criteria under which Behavioral Probe and Semantic Probe revisions may be proposed and the authorization requirements for committing them via Endure. Probe revisions are structural changes and follow the same authorization model as other Structural Consolidation Proposals.

**Integrity thresholds — mandatory:**

8. **Heartbeat threshold `T`** — the Pulse Counter value at which a Vital Check is triggered. The profile must define `T` or specify that it is defined at implementation time with a mandatory minimum value.
9. **Watchdog timeout** — the time-absolute deadline for synchronous skill execution before the SIL treats the condition as Critical. The profile must define the timeout value or specify the method by which it is computed from the declared skill manifest.
10. **Degraded accumulation threshold** — the number of consecutive Degraded states within a single session that triggers proactive SIL session token revocation. The profile must define this threshold.
11. **Reciprocal SIL Watchdog threshold and silence window** — the silence counter threshold and the window within which dual confirmation must occur for the Reciprocal SIL Watchdog escalation path.
12. **Context window critical threshold** — the context window utilization percentage at which the CPE signals the context window critical condition to the SIL.

**CPE topology — mandatory:**

13. **Declared CPE topology** — every profile must declare whether it supports Transparent CPE only, Opaque CPE only, or both. HACA-Core is architecturally constrained to Transparent CPE; this item is a formal declaration, not a choice.
14. **Health Flow polling interval** — the interval at which the SIL polls background skill results with status `running` through the Health Flow. This interval is distinct from the Heartbeat threshold `T` (item 8): `T` governs the Pulse Counter-based Vital Check frequency for all components, while the Health Flow polling interval governs only the SIL's monitoring of background skill execution results. The profile must define this interval or specify the method by which it is computed from the declared skill manifest.
15. **Passive Distress Beacon threshold N** — the number of consecutive failed Operator Channel delivery attempts that triggers Passive Distress Beacon activation. The profile must define N or specify that it is defined at implementation time with a mandatory minimum value.
16. **Health Flow signal intervals** — the intervals at which each component emits health signals to the SIL, and the intervals at which the SIL polls each component through the Health Flow. The profile must define these intervals or specify that they are defined at implementation time. The Reciprocal SIL Watchdog threshold (item 11) must be defined relative to these intervals — expressed in units of the defined polling interval so that the watchdog silence condition is deterministic.

**Temporal parameter ordering constraints — mandatory:** When defining items 8, 9, and 14, the profile must ensure: (1) the Health Flow polling interval (item 14) is shorter than the watchdog timeout (item 9), so the SIL can detect background skill failures before a concurrent synchronous skill's watchdog fires; (2) the watchdog timeout (item 9) is less than or equal to the expected duration of `T` cycles, so that execution stalls are detectable within the same Heartbeat period. A profile specification that violates these constraints is non-compliant.
